# PRD: AAA (Algorithmic Alpha Advisor)

> 최초 작성: 2026-02-21

---

## 1. 문제 정의 (Problem Statement)

### 배경
개인 투자자는 한국(KOSPI/KOSDAQ)과 미국(NYSE/NASDAQ) 주식 시장을 동시에 모니터링하기 어렵다. 두 시장은 거래 시간이 겹치지 않고, 분석에 필요한 데이터(가격, 수급, 거시경제, 공시 등)가 여러 출처에 분산되어 있으며, 이를 실시간으로 종합 판단하는 작업은 인간의 인지 용량을 초과한다.

### 핵심 문제
- **정보 과부하**: 실시간 체결, 호가, 투자자 수급, 공매도, 거시경제 지표 등을 동시에 추적 불가
- **감정 개입**: 수동 매매 시 손실 회피, 확증 편향 등 행동 편향이 의사결정을 왜곡
- **기회 손실**: 중요한 신호 발생 시 즉각 인지 불가 (취침, 업무 중 등)
- **반복 노동**: 매일 동일한 데이터 수집·분석 작업을 수동으로 반복

### 해결 방향
데이터 수집 → 머신러닝 분석 → 알림 → (반자동) 주문 실행의 파이프라인을 자동화하여, 개발자 본인이 더 나은 투자 의사결정을 내릴 수 있도록 지원한다.

## 2. 목표 (Goals)

### 핵심 목표
1. 한국·미국 주식 시장 데이터를 자동으로 수집하고 DB에 적재한다.
2. 머신러닝 모델로 매매 신호(등급 + 신뢰도)를 생성한다.
3. 신호 품질에 따라 계층화된 알림을 텔레그램으로 발송한다.
4. 텔레그램 명령을 통해 KIS API로 주문을 실행한다 (Phase 4).

### Phase별 목표

| Phase | 목표 |
|-------|------|
| Phase 1 | 데이터 파이프라인 완성, 안정적 수집 |
| Phase 2 | ML 모델 학습·추론, 매매 신호 생성 |
| Phase 3 | 알림 필터링, 텔레그램 발송 |
| Phase 4 | 반자동 주문 실행 |

서비스-Phase 매핑은 [5절](#5-서비스-구조)에 정의한다.

## 3. 성공 지표 (Success Metrics)

### Phase 1 (Collector)
- 수집 누락률 < 1% (장중 기준)
- 데이터 파이프라인 장애 시 자동 복구 (Fallback 체인 동작)
- 수집 지연 < 5초 (실시간 체결 기준)

### Phase 2 (Analyzer)
- 장 마감 후 전 종목 등급 확정 < [TBD]분 (일봉 수집 완료 후, Phase 2 중 실측 후 확정)
- 백테스트 기준 단순 매수·보유 전략 대비 샤프 비율 향상
- 신호 정확도(Precision) — Phase 2 초기 백테스트 후 이 절에서 기준 확정

### Phase 3 (Notifier)
- 타이밍 조건 충족 → 텔레그램 발송 지연 < 60초
- 알림 필터 오작동 0건 (필터가 차단했어야 할 알림이 통과한 경우. 쿨다운·확증 필터 효과 검증)
- 일일 리포트 정시 발송 100%

### Phase 4 (Trader)
- 텔레그램 주문 명령 처리 지연 < 3초
- 주문 실패 시 즉시 오류 알림

측정 방법 상세는 [TECHSPEC 11절](TECHSPEC.md#11-성능-지표-측정)에 정의한다.

## 5. 서비스 구조

AAA는 5개 서비스로 구성된 MSA(Micro Service Architecture)다.

| 서비스 | 역할 | Phase |
|--------|------|-------|
| aaa-collector | 데이터 수집 (KIS WebSocket/REST + 외부 API) | Phase 1 |
| aaa-analyzer | ML 분석, 매매 신호 생성 | Phase 2 |
| aaa-notifier | 알림 필터링 + 텔레그램 발송 | Phase 3 |
| aaa-trader | 텔레그램 명령 → KIS API 주문 실행 | Phase 4 |
| aaa-infra | Docker Compose 등 인프라 공통 관리 | 전체 |

서비스 간 통신 방식, 메시지 브로커, DB 분리 전략 등 기술적 상세는 [TECHSPEC 1.2절](TECHSPEC.md#12-서비스-간-통신-흐름)에 정의한다.

## 6. 분석 대상

### 6.1 대상 시장

- 한국: KOSPI, KOSDAQ
- 미국: NYSE, NASDAQ
- 두 시장 동일 비중으로 지원

### 6.2 ML 매매 신호 생성 대상

관심 종목을 ML 모델 적용 여부에 따라 분류한다. 데이터 수집은 전 종목 공통이다.

**ML 모델 대상 (개별 주식 + ETF)**

시장별(국내/해외) 전 종목(개별 주식 + ETF)을 각각 하나의 풀링 모델로 통합 학습한다. 종목 속성 피처(시가총액, 업종, 베타 등)로 종목을 구분하며, 추론 시 종목별 개별 예측을 생성한다. 모델 구조 상세(시장별 모델 수, 레이블 경계 등)는 [TECHSPEC 6.1절](TECHSPEC.md#61-모델-구조)에 정의한다.

모델링 적합 종목은 정량 기준으로 자동 등급 분류한다.

| 등급 | 의미 | 처리 |
|------|------|------|
| A | 충분한 데이터·유동성 | ML 추론 대상 |
| B | 보통 수준 | ML 추론 대상 |
| C | 데이터·유동성 부족 | ML 추론 제외 (데이터 수집만) |

정량 분류 기준(상장 기간, 거래대금 등)은 [TECHSPEC 6.5절](TECHSPEC.md#65-종목-등급-분류-로직)에 정의한다.

- 신규 종목 추가 시 정량 기준으로 등급 자동 확정, 즉시 적용
- 사용자가 텔레그램 명령으로 등급 수동 수정 가능 (예: C→A 상향 시 ML 추론 대상 포함, A→C 하향 시 ML 추론 대상 제외). 텔레그램 명령은 Phase 4에서 구현하며, Phase 4 이전에는 DB 직접 수정으로 대체
- 등급 변경은 다음 재학습 주기에 반영

**중복 ETF 처리**: 같은 거래소 + 같은 기초지수를 추종하는 ETF가 여러 개 존재할 경우, 대표 ETF 1개만 모델링하고 나머지는 데이터 수집만 수행한다. 대표 선정 기준은 [TECHSPEC 6.5절](TECHSPEC.md#65-종목-등급-분류-로직)에 정의한다.

**F등급 — ML 모델링 부적합**

| 대상 | 예시 | 제외 사유 |
|------|------|-----------|
| TDF | KODEX TDF2045액티브 | 글라이드패스에 따라 자산배분 자동 변경, 패턴 비일관 |
| 액티브 ETF | TIMEFOLIO Korea플러스배당액티브 | 운용역 재량으로 종목 변경, ML이 시장 패턴이 아닌 펀드매니저 패턴을 학습 |

F등급은 데이터 수집만 수행하고 ML 모델을 생성하지 않는다. 사용자가 등급을 수동 변경할 수 있다.

**ML 모델 미생성 대상**

순수 지수, 금리 지표(CD금리/머니마켓/SOFR 등)는 ML 모델을 생성하지 않고 다른 종목의 입력 데이터로만 활용한다 (제외 사유: [4절 비목표](#4-비목표-non-goals) 참조). 대상 목록과 분류 기준은 [TECHSPEC 7.1절](TECHSPEC.md#71-피처-카테고리-개요)에 정의한다.

종목별 등급 분류 상세는 [TECHSPEC 6.5절](TECHSPEC.md#65-종목-등급-분류-로직), 피처 파이프라인은 [7절](TECHSPEC.md#7-피처-엔지니어링)에 정의한다.

## 7. 데이터 수집 범위

### 7.1 KIS API 제약

KIS API 제약 (REST Rate Limit, WebSocket 구독 상한 등) 내에서 설계한다. 운영자 보유 KIS 계좌(앱키) 5개를 전제로 용량을 설계한다. 제약 수치 및 종목 수 상한은 [TECHSPEC 3.1절](TECHSPEC.md#31-kis-api-제약-및-수용-설계)에 정의한다.

### 7.2 KIS API 수집 데이터

**국내**: OHLCV (일봉), 실시간 체결·호가, 투자자별 매매동향, 프로그램 매매, 공매도, 신용잔고, 재무제표, 업종지수, 금리, 증시자금, 배당·증자, 투자의견, 뉴스 (제목만)

**해외**: OHLCV, 실시간 체결, 호가 (Level 1), 해외선물 (VX 포함), 배당·권리, 뉴스 (제목만)

수집 방식(REST/WebSocket)·주기 등 상세는 [TECHSPEC 3.2절](TECHSPEC.md#32-kis-api-국내-수집-상세), [3.3절](TECHSPEC.md#33-kis-api-해외-수집-상세)에 정의한다.

### 7.3 외부 API (Fallback 체인)

환율/VIX/Pre-After-Hours 등은 외부 무료 API로 수집하며, 장애 시 Fallback 체인을 적용한다. 상세는 [TECHSPEC 3.4절](TECHSPEC.md#34-외부-api-fallback-체인)에 정의한다.

### 7.4 기타 외부 API

| 데이터 | 출처 |
|--------|------|
| 미국 공매도 (일별) | FINRA Daily Short Volume |
| 미국 공매도 (반월) | FINRA Short Interest |
| 한국 공시 | DART OpenAPI |
| 한국 거시경제 | 한국은행 ECOS |
| 미국 거시경제 | FRED API |
| 한국 뉴스 | 네이버 뉴스 검색 API (Phase 2 중 도입) |
| 해외 뉴스 | GDELT (Phase 2 중 도입 여부 결정) |

상세 수집 방법 및 제한은 [TECHSPEC 3.5절](TECHSPEC.md#35-단일-소스-외부-api)에 정의한다.

### 7.5 데이터 보관 정책

- 일봉: 영구 보관
- 실시간 틱 원본: 비보관 (타이밍 평가 소비 후 폐기)
- 관심 종목 동기화: 1일 2회(한국·미국 장 시작 전), KIS API → DB. 관심 종목 목록(추가/삭제)은 KIS MTS에서만 관리하며, 시스템 내 목록 편집 불가. ML 등급 변경은 6.2절 참조

보관 정책 상세(관심 종목 동기화)는 [TECHSPEC 3.6절](TECHSPEC.md#36-데이터-보관-정책)에 정의한다.

시간외 데이터 수집·활용 상세는 [TECHSPEC 3.7절](TECHSPEC.md#37-시간외-데이터-수집-정책)에 정의한다.

### 7.6 과거 데이터 백필

ML 학습(Phase 2)에 필요한 과거 데이터를 Phase 1에서 사전 수집한다. 일봉 OHLCV, 수급, 거시경제 등 배치 수집 대상 데이터를 종목·지표별로 조회 가능한 최대 과거까지 소급 수집한다.

백필 전략 상세(방식, 완료 기준)는 [TECHSPEC 3.9절](TECHSPEC.md#39-과거-데이터-백필-전략)에 정의한다.

## 8. 매매 신호 정의

### 8.1 신호 등급

| 등급 | 의미 |
|------|------|
| STRONG_BUY | 강한 매수 신호 |
| BUY | 매수 신호 |
| HOLD | 보유 유지 |
| SELL | 매도 신호 |
| STRONG_SELL | 강한 매도 신호 |

각 신호에는 confidence (0.0 ~ 1.0) 값이 함께 생성된다. 모델은 5개 등급을 직접 출력하며, STRONG 등급은 더 높은 수익/손실이 예상될 때 부여된다. 등급 분류에 사용하는 수익률 범위 기준은 [TECHSPEC 6.1절](TECHSPEC.md#61-모델-구조)에 정의한다.

### 8.2 신호 방향 정의

신호를 방향성 기준으로 분류한다.

| 방향 | 포함 등급 |
|------|-----------|
| 매수 방향 | STRONG_BUY, BUY |
| 중립 | HOLD |
| 매도 방향 | SELL, STRONG_SELL |

HOLD는 "무행동" 신호로서, 매수/매도 어느 방향에도 속하지 않는 완충지대다.

### 8.3 시간대 (Time Horizon)

모든 ML 모델은 두 시간대로 분리하여 독립적으로 학습·추론한다.

| 시간대 | 예측 범위 | 용도 |
|--------|-----------|------|
| 단기 (20영업일) | ~1개월 | 스윙 트레이딩 신호 |
| 중기 (60영업일) | ~3개월 | 포지션/추세 신호 |

동일 시점에서 단기·중기 방향이 반대일 수 있으므로 (예: 단기 하락 + 중기 상승) 시간대별 독립 모델로 레이블 충돌을 방지한다.

## 9. 알림 전략

### 9.1 알림 Tier

신호 중요도에 따라 3단계로 계층화한다. 각 Tier별 확증·쿨다운·confidence 임계값 등 구체 수치는 [TECHSPEC 8.3절](TECHSPEC.md#83-방향-전환-유형별-확증쿨다운) 및 [8.5절](TECHSPEC.md#85-tier-분류-기준)에 정의하며, 운영 중 튜닝한다.

| Tier | 조건 | 원칙 |
|------|------|------|
| Tier 1 | STRONG 등급 + 높은 confidence | 즉시 발송, 같은 방향 반복만 쿨다운 적용 |
| Tier 2 | 방향 전환 | 전환 유형별 확증·쿨다운 비대칭 적용 (방향 전환 유형별 세분화는 [TECHSPEC 8.3절](TECHSPEC.md#83-방향-전환-유형별-확증쿨다운) 참조) |
| Tier 3 | 나머지 신호 변동 | 집계 알림 또는 일일 리포트로 라우팅 |

### 9.2 알림 필터 파이프라인

알림은 타이밍 평가·히스테리시스·확증·쿨다운·confidence·분류의 6단계 필터를 거친다. 각 필터의 구체 파라미터는 [TECHSPEC 8절](TECHSPEC.md#8-알림-필터-파이프라인)에 정의한다.

### 9.3 알림 채널

텔레그램 봇 2개를 1:1 DM 방식으로 운영한다. 한국/미국 시장 알림은 분리하지 않는다.

| 봇 | 용도 |
|------|------|
| 매매 봇 | 매매 신호 알림, 일일 리포트, 주문 실행 |
| 시스템 봇 | 장애 알림, 자원 경고, 자동 복구 결과 |

봇을 독립 토큰·커넥션으로 분리하여 매매봇 장애 시에도 시스템봇에 의한 시스템 알림은 정상 수신할 수 있도록 한다. 봇 분리 설계 상세는 [TECHSPEC 1.4절](TECHSPEC.md#14-텔레그램-채널-아키텍처)에 정의한다.

주문 명령어(`/buy`, `/sell`)는 Telegram User ID 화이트리스트로 권한을 제어한다.

### 9.4 시간외 알림 정책

시간외 데이터에 대해서는 매매 신호가 아닌 이벤트 기반 룰 필터로 "주의 환기" 알림만 발송한다.

| 조건 | 알림 유형 |
|------|-----------|
| 어닝 발표 후 갭이 임계치 이상 | 주의 환기 (매매 신호 아님) |
| 시간외 비정상 거래량 감지 | 주의 환기 |
| 그 외 시간외 가격 변동 | 알림 없음 (정규장 피처로만 흡수) |

구체 임계값(갭 크기, 거래량 배수 등)은 운영 초기 튜닝 후 [TECHSPEC 3.7절](TECHSPEC.md#37-시간외-데이터-수집-정책)에 확정한다.

## 10. 장애/복구 전략

### 10.1 자동 복구 원칙

| 원칙 | 설명 |
|------|------|
| 대상 한정 | 재연결, 토큰 갱신, API 재시도, Fallback 전환만 자동 복구 |
| 재시도 상한 필수 | 모든 자동 재시도는 횟수·시간 상한을 정의. 상한 초과 시 수동 개입으로 전환 |
| 부작용 금지 | 자동 복구에서 데이터 삭제·설정 변경은 금지 |
| 반복 감지 | 동일 자동 복구가 기준치 이상 반복 시 수동 개입 트리거 |

### 10.2 수동 개입 기준

다음 중 하나라도 해당하면 자동 복구를 중단하고 수동 개입으로 전환한다.

- 설계 변경이 필요한 장애 (한도 초과, Rate Limit 반복 위반)
- 데이터 무결성 확인이 필요한 장애 (DB 크래시 후 재시작)
- 자원 한계 장애 (디스크 풀, 메모리 부족)
- 자동 복구 상한 초과

### 10.3 안전 모드

자동 복구 상한 초과 시 운영자가 즉시 대응할 수 없는 상황(취침, 업무 중)을 전제로, 장애가 발생한 부분만 비활성화하고 나머지 정상 파이프라인은 계속 운영한다.

- 예: KIS WebSocket 재연결 실패 → 실시간 수집만 중단, REST 배치 수집 + ML 추론 + 알림은 정상 운영
- 안전 모드 진입·해제 이벤트는 시스템 채널로 알림
- 격리 단위, 상태 관리, 진입·해제 조건 상세는 [TECHSPEC 1.3절](TECHSPEC.md#13-안전-모드-safe-mode)에 정의한다.

### 10.4 장애 알림 기준

| 수준 | 조건 |
|------|------|
| 즉시 알림 | 서비스/DB 재시작, 재연결 최종 실패, 자원 임계치 초과, Fallback 소진, 이상값 감지, 안전 모드 진입 |
| 일일 리포트 | 자동 복구 횟수 요약, 수집 누락률, Fallback 전환 내역 |
| 알림 없음 | 재시도 중 일시 실패(최종 성공), 장외 시간 스트림 중단, 예정된 점검 |

장애 알림은 시스템 채널로 발송한다. 텔레그램 자체 장애 시에는 로컬 로그에 보존하고, 복구 후 장애 기간 이벤트를 요약 발송한다. 텔레그램 장애 기간 중 운영자 인지는 수동 확인에 의존한다.

### 10.5 자원 모니터링

NAS 자원에 대해 단계별 임계치 알림을 설정한다. 구체 임계값은 [TECHSPEC 10.5절](TECHSPEC.md#105-자원-모니터링-임계값-nas)에 정의한다.

| 자원 | 모니터링 이유 |
|------|--------------|
| 디스크 | 일봉·수급·거시경제 등 영구 보관 데이터 누적으로 장기적 소진 불가피 |
| RAM | analyzer Python 추론 프로세스 메모리 집중 |
| CPU | 장 시작 직후 수집·추론 동시 부하 |

## 11. 주문 실행 (Phase 4)

### 11.1 주문 트리거

두 가지 방식을 모두 지원한다.

| 방식 | 설명 |
|------|------|
| 알림 인라인 버튼 | STRONG 등급 알림에 [매수] / [매도] 버튼 포함, 탭하면 주문 확인 단계로 진입 |
| 직접 명령어 입력 | 텔레그램에서 명령어로 주문 (예: `/buy 삼성전자 10`) |

### 11.2 주문 유형

| 유형 | 설명 |
|------|------|
| 시장가 | 즉시 체결 |
| 지정가 | 지정 가격에 주문 |

### 11.3 주문 흐름

주문은 사용자 확인 후 실행하는 반자동 방식이다. 실행 흐름 상세는 [TECHSPEC 9.1절](TECHSPEC.md#91-주문-흐름-7단계)에 정의한다.

### 11.4 범위

- 주문 실행(매수/매도)만 지원, 정규장에서만 실행 (시간외 주문 미지원)
- 보유 현황·손익 조회는 KIS MTS에서 직접 확인 (시스템 미지원)
