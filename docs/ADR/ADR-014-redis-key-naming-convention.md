# ADR-014: Redis 키 네이밍 컨벤션 — 기능 우선(Function-first) 네임스페이스 채택

- **상태**: 승인
- **일자**: 2026-03-21
- **범위**: aaa-infra

---

## 맥락

단일 Redis 인스턴스를 여러 서비스(collector, analyzer, notifier, trader)가 공유하는 MSA 구조다. 키 패턴은 TECHSPEC 5.1~5.4에 각 섹션별로 정의되어 있으나, 키를 구성하는 네이밍 규칙 자체(구분자, 단어 표기법, 네임스페이스 분류 축)는 명시된 적이 없었다.

규칙 없이 키가 추가되면 다음 문제가 발생한다.

- 구분자 혼용(`:`과 `-`이 섞임)으로 패턴 매칭이 어려워짐
- 네임스페이스 분류 기준이 담당자마다 달라져 일관성이 무너짐
- TTL 정책이나 일괄 조회 시 패턴 기반 필터링이 불가능해짐

---

## 결정

### 1. 구분자 규칙

- **세그먼트 간 구분**: 콜론(`:`) 사용
  - Redis 공식 문서 권장 방식이며 사실상의 업계 표준
- **복합 단어 내부**: 언더스코어(`_`) 사용
  - `safe_mode`, `api_call`, `tick_price` 등

### 2. 기능 우선(Function-first) 네임스페이스

최상위 네임스페이스는 데이터의 기술적 역할(기능)을 기준으로 분류한다.

| 네임스페이스 | 역할 |
|-------------|------|
| `cache:` | 외부 API 응답, 반복 조회 데이터 캐싱 |
| `stream:` | Redis Streams 기반 서비스 간 이벤트 버스 |
| `metric:` | API 호출 횟수 등 경량 지표 |
| `safe_mode:` | 안전 모드 상태 플래그 (예외, 아래 설명) |

### 3. `safe_mode` 독립 네임스페이스 유지

`safe_mode:{alias}`는 캐시·스트림·메트릭 중 어느 범주에도 속하지 않는 상태 플래그다. 기능 범주로 묶으면(`flag:safe_mode:...`) 키만 길어지고 실질적 분류 이득이 없으므로, 최상위 네임스페이스로 독립 유지한다.

---

## 검토한 대안

### 대안 1: 서비스 우선(Service-first) 패턴 — 기각

`collector:cache:token:{alias}` 형태로 서비스명을 최상위에 두는 방식.

| 항목 | 내용 |
|------|------|
| 장점 | 키만 보고 어느 서비스 소유인지 즉시 파악 가능 |
| 단점 | 기능별 일괄 조회/TTL 정책 적용이 어려움. 단일 Redis 인스턴스 환경에서 서비스 격리 효과가 없음 |

현재 환경은 ACL로 접근 제어를 하며, 기능 단위로 TTL을 일괄 관리할 필요가 더 크다. 기각.

### 대안 2: `safe_mode`를 `flag:` 네임스페이스로 범주화 — 기각

`flag:safe_mode:{alias}` 형태로 상태 플래그를 하나의 범주로 묶는 방식.

| 항목 | 내용 |
|------|------|
| 장점 | 모든 키가 기능 네임스페이스로 시작하는 일관성 |
| 단점 | `safe_mode` 외 다른 상태 플래그 추가 가능성이 낮아 과잉 설계. 키만 길어지는 결과 |

기각.

---

## 결과

- 신규 키 추가 시 세그먼트 구분자는 콜론(`:`), 복합 단어 내부는 언더스코어(`_`)를 따른다.
- **트레이드오프**: `safe_mode`만 분류 축(상태 플래그)이 다르지만, 성격상 자연스러운 구분이다. 다른 최상위 네임스페이스(`cache:`, `stream:`, `metric:`)는 데이터 기술 역할 기준이고, `safe_mode:`는 도메인 상태 기준이다. 이 불일치는 의도된 예외로 문서화한다.
