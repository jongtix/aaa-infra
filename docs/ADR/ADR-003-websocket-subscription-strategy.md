# ADR-003: WebSocket 구독 대상 동적 관리 및 Phase 1 임시 정책

- **상태**: 승인
- **일자**: 2026-02-28
- **관련 문서**: [TECHSPEC 3.1절 — KIS API 제약 및 수용 설계](../TECHSPEC.md#31-kis-api-제약-및-수용-설계), [PRD 7.1절 — KIS API 제약](../PRD.md#71-kis-api-제약)

---

## 맥락

KIS API는 WebSocket 구독 건수에 물리 상한이 있다.

| 항목 | 수치 |
|------|------|
| WebSocket 세션 | 계좌(앱키)당 1세션, 보유 5계좌 = 최대 5세션 |
| 세션당 구독 | 41건 |
| 최대 구독 건수 | 5 × 41 = 205건 |
| 1종목 소모 건수 | 체결(H0STCNT0) + 호가(H0STASP0) = 2건 |
| **실질 최대 종목 수** | **205 ÷ 2 = 102종목** |

관심 종목(A·B등급)은 수백 종목에 달할 수 있다. 전부 구독하면 물리 상한을 초과한다. 따라서 "어떤 종목을 WebSocket으로 구독할 것인가"를 결정해야 한다.

Phase 1에는 ML 모델이 없다. ML 등급 기반 선정이 불가능한 상태에서 WebSocket → Redis Streams 경로를 검증해야 한다.

---

## 결정

**정규 운영 (Phase 2 이후)**: ML 등급이 BUY, STRONG_BUY, SELL, STRONG_SELL인 종목만 구독한다. 장 시작 전 MySQL에서 대상 종목 목록을 조회하여 동적으로 결정한다. HOLD 및 C/F등급 종목은 구독 제외다.

**Phase 1 임시 정책**: A·B등급 전 종목을 구독 대상으로 한다. 102종목 상한 초과 시 상한까지 임의 절삭한다. Phase 2에서 ML 등급 기반 선정으로 전환한다.

---

## 검토한 대안

### 대안 1: 관심 종목 전체 구독 시도 (초과 시 에러 처리)

A·B등급 전 종목에 구독 요청을 보내고, KIS API 에러 응답부터 구독을 중단한다.

| 항목 | 내용 |
|------|------|
| 장점 | 구현 단순 |
| 단점 (결정적) | KIS API 에러 처리 방식이 명확하지 않으며, 초과 구독 시도 자체가 세션 안정성에 영향을 줄 수 있음 |
| 단점 | 어떤 종목이 누락되었는지 예측 불가 (순서 의존) |

**탈락 이유**: 에러 동작이 불확실하고 누락 종목 비결정적

### 대안 2: 관심 종목 수 자체를 102개 이하로 제한

등급 선정 기준을 조정하여 A·B등급 종목 수가 항상 102개 이하가 되도록 한다.

| 항목 | 내용 |
|------|------|
| 장점 | 상한 초과 문제 원천 제거 |
| 단점 (결정적) | ML 분석 대상(REST 배치 수집)이 WebSocket 물리 제약에 종속됨 — REST와 WebSocket의 범위가 서로 독립적이어야 한다는 설계 원칙 위반 |
| 단점 | 유동성 좋은 종목을 강제 제외해야 할 수 있음 |

**탈락 이유**: REST 수집 범위와 WebSocket 구독 범위의 결합도가 불필요하게 높아짐

### 대안 3: ML 등급 기반 동적 구독 (현재 선택)

ML 추론 결과(BUY/SELL 등급)를 구독 기준으로 삼는다. HOLD 및 C/F등급은 구독하지 않는다.

| 항목 | 내용 |
|------|------|
| 장점 | REST 배치 수집(일봉·수급·재무 등) 범위가 WebSocket 제약에 종속되지 않음 |
| 장점 | BUY/SELL 등급은 통상 수~수십 종목 — 102종목 상한 내 충분히 수용 |
| 장점 | 실시간 체결·호가 데이터가 실제 매매 판단에 쓰이는 종목에만 집중 |
| 단점 | Phase 1에서 ML 모델 미존재 → 임시 정책 필요 |

**Phase 1 임시 정책 선택 근거**:
- A·B등급은 거래량 기준 상위 종목으로 유동성이 높아 WebSocket 검증용 테스트 데이터로 적합
- Phase 1 목적은 WebSocket → Redis Streams 경로 검증이므로, 구독 종목의 정확한 선별보다 경로 동작 확인이 우선
- 102종목 상한 초과 시 임의 절삭은 검증 목적에 충분하며, Phase 2 전환 시 정책이 교체됨

---

## 결과

- WebSocket 구독 전략은 ML 등급 기반 동적 선정으로 확정된다.
- REST 배치 수집 범위(전 A·B등급 종목)와 WebSocket 구독 범위(BUY/SELL 등급만)가 분리되어, ML 분석 입력 데이터가 WebSocket 물리 제약에 종속되지 않는다.
- Phase 1에서는 임시 정책(A·B등급 전 종목, 초과 시 절삭)으로 운영하고, Phase 2 착수 시 ML 등급 기반으로 전환한다.
- 구독 대상 목록은 장 시작 전 MySQL 조회로 결정하며, 장중 실시간 변경은 하지 않는다 ([TECHSPEC 3.1절](../TECHSPEC.md#31-kis-api-제약-및-수용-설계) 참조).
