# ADR-020: 배포 중 다운타임 허용 — blue/green 배포 미채택

- **상태**: 승인
- **일자**: 2026-04-08

---

## 맥락

[ADR-018](ADR-018-github-actions-cd.md)에서 채택한 배포 명령은 다음과 같다.

```bash
docker compose up -d --wait --wait-timeout 180 collector
```

`docker compose up`은 컨테이너를 재생성(stop → create → start)한다. 이전 컨테이너가 중단된 후 새 컨테이너가 `healthy` 상태가 될 때까지 collector 서비스가 down 상태에 놓인다. JVM cold start를 포함하면 수십 초~수분의 다운타임이 발생한다.

collector는 향후 KIS WebSocket을 통한 실시간 틱 데이터 수집을 담당할 예정이나, 현재 Phase 1 시점에서는 미구현 상태다.

---

## 결정

`docker compose up -d --wait collector` 방식의 배포 중 다운타임을 허용된 트레이드오프로 수용한다. blue/green 배포는 채택하지 않는다.

---

## 검토한 대안

### 대안 1: blue/green 배포 — 기각

| 항목 | 내용 |
|------|------|
| 장점 | 배포 중 서비스 중단 없음, 실시간 데이터 수집 누락 방지 |
| 단점 | Docker Compose 환경에서 별도 Compose 파일 또는 네트워크 전환 로직 필요. 헬스체크 통과 후 트래픽 전환 자동화 구현 복잡도 증가. 1인 프로젝트 + Phase 1 단계에서 운영 복잡도 대비 현재 얻는 이득이 없음 |

기각. WebSocket 실시간 수집이 미구현된 현 시점에서 실시간 데이터 손실 자체가 발생하지 않는다. 구조적 복잡도를 도입할 근거가 없다.

### 대안 2: 현행 방식 유지 + 다운타임 허용 (채택)

| 항목 | 내용 |
|------|------|
| 장점 | 배포 파이프라인 단순, 별도 인프라 변경 불필요 |
| 단점 | 배포 시 수십 초~수분 다운타임 발생 |

채택.

---

## 결과

배포 파이프라인은 ADR-018 구조를 그대로 유지한다. 다운타임은 허용된 트레이드오프로 수용한다.

**재검토 트리거**: 다음 중 먼저 오는 시점에 이 결정을 재검토한다.

1. WebSocket 실시간 수집 구현 이후, 재연결 + REST 백필로 커버 불가능한 데이터 누락이 확인되는 시점
2. Phase 3(notifier) 진입 시점
