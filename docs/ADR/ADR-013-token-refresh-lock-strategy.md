# ADR-013: 토큰 재발급 동시성 제어 — JVM ReentrantLock 채택

- **상태**: 승인
- **일자**: 2026-03-15
- **범위**: aaa-collector

---

## 맥락

KIS Open API 토큰 관리에서 동시 401 응답 수신 시 중복 재발급을 방지해야 한다.

WebSocket과 REST 클라이언트가 동시에 동일 계좌의 유효 토큰을 요청하던 중 모두 401을 수신하면, 락 없이 구현할 경우 두 클라이언트가 각각 토큰 재발급 요청을 KIS Open API에 전송한다. 이는 다음 문제를 유발한다.

- **KIS API 중복 호출**: 불필요한 API 호출 발생
- **레이스 컨디션**: 먼저 발급된 토큰이 직후 발급된 토큰으로 덮어쓰여 무효화될 수 있음

계좌는 최대 5개이며 각각 독립적인 토큰을 보유하므로, 계좌별로 독립적인 락이 필요하다.

---

## 결정

### 동시성 제어: 계좌별 JVM ReentrantLock

계좌별로 `ReentrantLock` 인스턴스를 관리하는 방식을 채택한다.

- 락 범위: 토큰 재발급 로직 내부
- 락 단위: 계좌 단위 (계좌 식별자를 키로 하는 락 맵)
- 토큰 조회 인터페이스는 변경하지 않는다.

---

## 검토한 대안

### 대안 1: JVM 내 ReentrantLock (계좌별) — **채택**

| 항목 | 내용 |
|------|------|
| 장점 | 외부 의존성 없음, 구현 단순, 디버깅 용이, 네트워크 왕복 비용 없음 |
| 단점 | JVM 프로세스 범위로 제한 — 다중 인스턴스 배포 시 동시성 보장 불가 |

### 대안 2: Redis 분산 락 (SET NX + TTL) — 기각

| 항목 | 내용 |
|------|------|
| 장점 | 다중 인스턴스 환경에서도 동시성 보장 |
| 단점 | Redis 네트워크 왕복 비용 추가, TTL 튜닝 필요, 네트워크 장애 시 데드락 가능성 |

현재 배포 환경은 UGREEN DXP2800 NAS 단일 컨테이너다. 단일 인스턴스에서 분산 락은 불필요한 복잡성과 Redis 네트워크 비용을 추가한다. 기각.

---

## 결과

- 토큰 발급은 하루 최대 5건(계좌 수) 수준의 저빈도 작업이다. ReentrantLock과 Redis 분산 락 간 성능 차이는 무의미하다.
- **트레이드오프**: ReentrantLock은 JVM 프로세스 범위로 제한된다. 현재 단일 컨테이너 배포 전제에서는 이 제약이 문제되지 않는다.
- **전환 조건**: 다중 인스턴스 배포가 결정되면 Redis 분산 락(SET NX + TTL)으로 전환한다.
  - 전환 영향 범위: 토큰 재발급 서비스 내부 락 획득/해제 로직만 교체
  - 토큰 조회 인터페이스 변경 없음 (호출자 수정 불필요)
