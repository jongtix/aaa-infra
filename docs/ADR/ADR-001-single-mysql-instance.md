# ADR-001: 단일 MySQL 인스턴스 (서비스별 DB 분리 안 함)

- **상태**: 승인
- **일자**: 2026-02-28
- **관련 문서**: [TECHSPEC 4절 — DB 스키마](../TECHSPEC.md#4-db-스키마)

---

## 맥락

AAA는 4개 서비스(collector, analyzer, notifier, trader)로 구성된 MSA다. MSA 표준 패턴은 서비스별 DB를 물리적으로 분리하여 독립 배포 및 장애 격리를 보장한다.

그러나 운영 환경은 UGREEN DXP2800 NAS (Intel N100, 8GB RAM, 16GB 확장 예정)이며, 1인 개인 프로젝트다. 이 두 가지 제약이 DB 전략에 직접 영향을 준다.

또한 analyzer는 ML 피처 계산을 위해 collector가 저장한 테이블을 JOIN해야 한다. 서비스 간 HTTP 호출 없이 직접 DB 접근하는 것이 가장 단순하고 성능상 유리하다.

---

## 결정

**단일 MySQL 8.4 인스턴스 + 단일 DB** 구조를 채택한다. 서비스별 물리 분리 대신, MySQL 사용자와 권한으로 논리적 격리를 구현한다.

보완 조치 ([TECHSPEC 4절](../TECHSPEC.md#4-db-스키마) 접근 보안 참조):
- 서비스별 전용 MySQL 사용자 생성 + 최소 권한 원칙 (예: collector는 자신의 테이블에 INSERT/SELECT만)
- `order_log`, `notification_log` 테이블: 모든 애플리케이션 사용자에게 UPDATE/DELETE 미부여 (INSERT-ONLY)
- root 계정 원격 접속 금지
- 포트(3306) 호스트 바인딩 금지 — Docker 내부 네트워크에서만 접근

## 검토한 대안

### 대안 1: 서비스별 MySQL 인스턴스 분리 (MSA 표준)

각 서비스가 독립 MySQL 컨테이너를 운영한다.

| 항목 | 내용 |
|------|------|
| 장점 | 완전한 장애 격리, 독립 스케일링, 서비스별 스키마 자율성 |
| 단점 (결정적) | 인스턴스당 최소 ~650MB RSS (innodb_buffer_pool 기준). 4개면 ~2.6GB — NAS 8GB에서 다른 서비스 운영 불가 |
| 단점 | analyzer의 collector 데이터 접근을 위해 별도 API 또는 데이터 복제 레이어가 필요 |
| 단점 | 4개 인스턴스 백업·모니터링·버전 관리 부담 (1인 운영 한계) |

**탈락 이유**: NAS 메모리 예산 초과 + 1인 운영 복잡도 과다

### 대안 2: 동일 인스턴스, 서비스별 논리 DB 분리

동일 MySQL 인스턴스 내에서 `aaa_collector`, `aaa_analyzer` 등 DB를 분리한다.

| 항목 | 내용 |
|------|------|
| 장점 | 메모리 단일 인스턴스 유지, 논리적 분리감 |
| 단점 | MySQL은 DB 간 GRANT가 복잡하며, cross-database JOIN 시 `db.table` 명시 필요 |
| 단점 | 실질적 격리 효과가 사용자·권한 분리와 거의 동일하면서 관리 복잡도만 증가 |

**탈락 이유**: 단일 DB + 사용자 권한 분리 대비 실익 없음

### 대안 3: 단일 인스턴스 + 사용자·권한 분리 (현재 선택)

단일 인스턴스, 단일 DB. 서비스별 MySQL 사용자와 테이블 단위 권한으로 접근을 통제한다.

| 항목 | 내용 |
|------|------|
| 장점 | 메모리 ~650MB로 단일 인스턴스 운영 |
| 장점 | analyzer가 collector 테이블을 직접 JOIN 가능 — ML 피처 계산에 최적 |
| 장점 | 백업·모니터링·버전 관리 단일화 |
| 단점 | 한 서비스의 스키마 오류가 전체 DB에 영향을 줄 수 있음 |
| 단점 | 서비스 간 DB 레벨 격리가 불완전 (권한 설정 실수 시 접근 가능) |

---

## 결과

- MySQL 8.4 단일 인스턴스 1개로 전체 서비스 데이터를 관리한다.
- 물리 격리 대신 사용자·권한 격리로 MSA의 책임 분리 원칙을 부분적으로 달성한다.
- NAS 메모리 예산 ([TECHSPEC 10.3절](../TECHSPEC.md#103-docker-compose-구성)) 내에서 운영 가능하다.
- 향후 서비스 규모 증가 또는 운영 환경 업그레이드 시 인스턴스 분리 재검토가 필요하다.
