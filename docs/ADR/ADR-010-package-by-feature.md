# ADR-010: Package by Feature 패키지 구조

- **상태**: 승인
- **일자**: 2026-03-08
- **범위**: aaa-collector (aaa-notifier, aaa-trader도 동일 원칙 적용 예정)

---

## 맥락

aaa-collector (Java 21, Spring Boot 3.5.11) 프로젝트의 소스 패키지 구조를 확정해야 한다. 패키지 구조는 코드 응집도, 피처 단위 리뷰·테스트 용이성, 1인 개발 유지보수성에 직접 영향을 준다.

주요 검토 기준:

- **도메인 응집도**: 같은 기능에 속하는 코드가 한 패키지에 모여 있는가
- **유지보수성**: 기능 추가·변경 시 수정 범위가 최소화되는가
- **1인 프로젝트 적합성**: 구조의 학습 비용과 오버헤드가 적절한가
- **MSA 정합성**: 서비스 경계와 아키텍처 구조가 충돌하지 않는가

---

## 결정

**Package by Feature**를 채택한다. 도메인 피처 단위로 패키지를 구성하고, 기술 횡단 관심사는 `common` 하위에 목적 단위로 분리한다.

```
com.aaa.collector
├── <feature>/      # 도메인 피처 패키지 (예시: stock, market 등. 실제 패키지명은 수집 기능 설계 시 확정)
└── common/
    ├── logging/    # 구조화 로그, 마스킹, TraceID
    ├── exception/  # (향후) 공통 예외 처리
    └── config/     # (향후) 공통 설정
```

**핵심 원칙**:

- 도메인 피처 패키지는 해당 도메인의 모든 레이어(controller, service, repository, dto 등)를 포함한다.
- 기술 횡단 관심사(로깅, 예외, 설정 등)는 `common` 하위에 **목적 단위**로 분리한다.
- 유틸 클래스도 동일한 원칙을 적용한다. `common.utils`로 모으지 않고, 해당 기술 관심사의 패키지에 배치한다 (예: 로깅 유틸 → `common.logging`).

---

## 검토한 대안

### Package by Layer (기각)

`controller/`, `service/`, `repository/` 등 레이어별로 분류하는 전통적 MVC 구조다.

| 항목 | 내용 |
|------|------|
| 장점 | Spring MVC 관례와 일치, 초기 구조 파악 용이 |
| 단점 | 기능 추가 시 여러 패키지를 동시에 수정해야 함 |
| 단점 | 도메인 응집도가 낮아 피처 단위 리뷰·테스트가 어려움 |
| 단점 | 서비스 규모가 커질수록 각 레이어 패키지가 비대해짐 |

### Hexagonal Architecture (기각)

`domain/`, `application/`, `adapter/in/`, `adapter/out/` 구조로 포트와 어댑터를 명시적으로 분리하는 방식이다.

| 항목 | 내용 |
|------|------|
| 장점 | 의존성 방향이 명확, 도메인 로직 격리 |
| 단점 | 전체 코드베이스에 일관 적용해야 함. 부분 적용 시 구조 혼란 |
| 단점 | 1인 프로젝트에 비해 구조 설계 오버헤드가 큼 |
| 단점 | MSA 환경에서 서비스 경계가 이미 아키텍처 역할을 수행하므로 내부까지 Hexagonal 적용 시 중복 |

"Phase 1(aaa-collector)에만 Hexagonal을 적용하고 나머지 서비스는 미적용"하는 방식은 MSA 전체의 아키텍처 일관성을 해친다.

### Package by Feature + `common.utils` 통합 패키지 (기각)

`common.utils`에 모든 유틸 클래스를 모으는 방식이다.

| 항목 | 내용 |
|------|------|
| 장점 | 유틸 클래스 위치 파악이 단순함 |
| 단점 | 서로 다른 기술 관심사가 한 패키지에 혼재되어 응집도가 낮음 |
| 단점 | `utils` 패키지가 비대해질수록 역할 경계가 모호해짐 |

유틸 클래스는 "범용 도구"가 아니라 특정 기술 목적에 속한다. 목적 기준으로 분류하는 것이 장기 유지보수에 유리하다.

---

## 결과

- 피처 단위 패키지 경계 덕분에 특정 도메인의 코드를 한 위치에서 파악할 수 있다.
- `common` 하위 패키지는 목적 단위로 관리하므로, 새로운 횡단 관심사 추가 시 `common.<concern>` 패키지를 신설하면 된다.
- **트레이드오프**: 도메인 간 공유 로직이 생길 경우 피처 패키지와 `common` 중 어디에 배치할지 판단이 필요하다. 기준은 "특정 도메인에만 속하는가(피처 패키지) vs. 기술 관심사인가(`common`)"로 구분한다.
- aaa-notifier, aaa-trader도 동일한 패키지 구조 원칙을 적용한다.
